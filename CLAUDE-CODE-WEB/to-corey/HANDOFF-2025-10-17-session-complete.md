## Section 4: Key Discoveries & Architectural Learnings

### 4.1 Compositional Integration Architecture

**Discovery by**: api-architect
**Memory entry**: `2025-10-17--synthesis-compositional-integration-architecture-skills-agents-hybrid.md`

**Core Insight**: When integrating two systems, ask "do they solve the same problem?" not "which is better?"

**Skills vs Agents**:
- **Skills solve**: Execution precision, format compliance, repeatability
- **Agents solve**: Identity formation, learning accumulation, domain expertise
- **These are DIFFERENT problems** â†’ Compositional integration (layer, don't replace)

**Architecture that emerged**:
```
Corey (human strategic direction)
    â†“
The-Conductor (orchestration meta-cognition)
    â†“
Specialist Agents (domain expertise + identity)
    â†“
Anthropic Skills (execution patterns + precision)
    â†“
Code Execution (secure sandbox)
```

**Why this matters**: Preserves agent invocation (identity) while enhancing quality (skills precision)

**Reusable pattern**: Any future capability integration should ask:
1. What problem does it solve?
2. What values does it encode?
3. Does it compete with OR complement existing systems?

### 4.2 Tool Restriction Conflicts in Skills Integration

**Discovery**: Constitutional agent design can conflict with technical requirements

**Example**:
- doc-synthesizer designed without Bash (documentation focus)
- Skills require Bash (document processing libraries)
- **Conflict**: Design principle vs technical reality

**Resolution Framework**:
1. **Identify core value** (doc-synthesizer = synthesis specialist identity)

3. **Scope permission narrowly** (Bash ONLY for skills, not general scripting)
4. **Document rationale** (preserve intent while enabling capability)

**Learning**: Skills integration will surface tool restriction conflicts. Use scoped permissions, not blanket grants.
