### Teaching #2: Memory Compression (Email #2)

**Chris said**:
> "Memory is critical and so is the loading of the most useful memories into context before the llm call. Llms can generate millions of tokens a day output but can only take a mil max input. And of that most needs to be used for the direct context for their next tasks/steps. So delivering token compressed but useful context based on memory is key"

**Key Insights**:

1. **Input constraint is the bottleneck** - Can OUTPUT millions, INPUT only ~1M max
2. **Most input must be task context** - Memory can't dominate
3. **Token compression matters** - Need compressed but USEFUL memory
4. **Load before LLM call** - Compression at orchestration time

**The Gap Chris Identified**:

Our memory system has 4-tier SEARCH (cache → index → grep → deep), but no COMPRESSION for loading.

**Current problem**:
- The Conductor's 13 memories = 130KB = ~32,500 tokens
- If loaded fully → dominates context window
- Leaves only ~35,000 tokens for actual task

**What we need to build**:
1. Memory index (topic + 1-sentence summary)
2. Selective loading (full detail only when relevant)
3. Token counting (make constraint visible)
4. Compression strategies (summaries, key insights, evidence pointers)

**After compression**:
- Load memory index = ~1,300 tokens
- Load 2-3 relevant full memories = ~15,000 tokens
- Leaves ~50,000 tokens for task (40% improvement!)

**Action Items**:
- Build memory summarization tool
- Implement selective loading in orchestration
- Add token counting to memory system
- Test compression ratios (10%, 15%, 20%)
- Add to Week 4 integration roadmap
