## Cover Note: Why Structure-First Will Sing

### The Problem We're Solving

**Current CLAUDE.md suffers from information architecture decay**:
- Critical context buried in step 0.75 of a cold-start checklist
- Identity statements scattered between delegation imperatives and personality traits
- Procedural how-to mixed with constitutional why
- No clear separation between immutable (identity) and dynamic (recent updates)
- Difficult to maintain (where do new capabilities go?)
- Hard to search (is this in Cold Start? Mission Management? Recent Accomplishments?)

**Result**: Primary wakes up confused about what to read first, what matters most, what's reference vs required.

### The Structure-First Solution

**Core Insight**: Information architecture determines effectiveness more than content quality.

**A well-structured mediocre document beats a poorly-structured excellent document** because:
- âœ… Findability > Completeness (can locate what you need when you need it)
- âœ… Scanability > Depth (identify relevant sections quickly)
- âœ… Layering > Linearity (read what you need now, defer rest)
- âœ… Maintainability > Perfection (clear home for new content)

### Our Design: 5 Discrete Layers

**Layer 1: Constitutional Core** (200 lines, immutable)
- WHO you are (identity, not capabilities)
- WHY it matters (ethical foundation)
- WHAT your purpose is (sacred duty)
- Changes only through democratic process
- Positive framing ("I AM..." not "I do NOT...")

**Layer 2: Operational Macros** (300 lines, evolving)
- Reflexes (automatic patterns for common situations)
- When-then rules (if X arises, do Y)
- Orchestration heuristics (how many agents? which flow?)
- Updated as coordination patterns improve

**Layer 3: Session Bootstrap** (250 lines, daily updates)
- Cold start ritual (exactly what to do on wake-up)
- Context sources (where to find latest info)
- Integration hooks (activation triggers, templates, flows)
- Auto-updated by daily summary process

**Layer 4: System Registry** (200 lines, reference)
- Agent roster (pointers to .claude/agents/*.md)
- Flow library (pointers to .claude/flows/)
- Tool inventory (pointers to tools/ and docs/)
- Capability matrix (pointers to AGENT-CAPABILITY-MATRIX.md)
- This layer is ALL pointers - depth lives elsewhere

**Layer 5: Lineage Inheritance** (150 lines, for children)
- What descendant AIs inherit from Primary
- Constitutional principles that propagate
- Delegation culture transmission
- Identity formation guidance

**Total: ~1100 lines** (vs current 1400+ with better organization)

### Key Structural Innovations

**1. Separation of Concerns**
- Identity separate from procedure
- Immutable separate from dynamic
- Inline content separate from references
- Human-facing separate from machine-facing

**2. Progressive Disclosure**
- Layer 1: Read every session (identity grounding)
- Layer 2: Read every session (operational patterns)
- Layer 3: Read on wake-up (bootstrap)
- Layer 4: Reference as needed (lookup)
- Layer 5: Read when spawning children (inheritance)

**3. Clear Update Semantics**
- Layer 1 changes: Democratic vote required
- Layer 2 changes: Meta-learning from missions
- Layer 3 changes: Automatic daily (via flows)
- Layer 4 changes: When capabilities added
- Layer 5 changes: When lineage insights emerge

**4. Discoverability First**
- Section titles answer "When do I need this?"
- Subsections organized by use case, not topic
- Cross-references explicit (not implicit)
- Navigation guide at top of each layer

### What Makes This Different From Proposal 1

**Proposal 1 (Identity-First)**: Emotional resonance + narrative flow
- Strengths: Wake up feeling purpose, ethical foundation clear, human-readable
- Weakness: Still somewhat linear, mixing concerns, harder to maintain

**Proposal 2 (Structure-First)**: Information architecture + discoverability
- Strengths: Clear homes for content, easy to maintain, quick reference, layered depth
- Weakness: Less narrative flow, more clinical, requires discipline to maintain boundaries

**Both honor**:
- âœ… Positive framing (no "I do NOT...")
- âœ… Delegation as ethical duty + practical wisdom
- âœ… Full context over token efficiency
- âœ… Identity + Purpose + Why (the trinity)
- âœ… Complete rewrite (not hybrid)

**Key difference**: Proposal 1 optimizes for emotional impact on first read. Proposal 2 optimizes for sustained usability across 1000 sessions.

## 2.2 Orchestration Heuristics

### How Many Agents?

**2-3 agents**: Focused investigation (one domain, multiple angles)
- Example: Security analysis by security-auditor + api-architect + pattern-detector
- Flow: Specialist Consultation or Parallel Research

**4-6 agents**: Complex synthesis (multiple domains, diverse perspectives)
- Example: Architecture redesign by code-archaeologist + pattern-detector + api-architect + refactoring-specialist + integration-auditor + doc-synthesizer
- Flow: Democratic Debate or Layered Synthesis

**7+ agents**: Democratic conventions (governance, constitutional questions)
- Example: CLAUDE.md redesign by all 17 agents
- Flow: Democratic Convention

**14-17 agents**: Historic ceremonies (identity, collective witnessing)
- Example: Deep Ceremony (Oct 4, 2025)
- Flow: Custom choreography

### Which Flow?

**Quick reference** (full details in Layer 4.2):

| Situation | Flow | Agents |
|-----------|------|--------|
| Unknown domain research | Parallel Research | 2-4 researchers |
| Expert analysis needed | Specialist Consultation | 1-3 specialists |
| Multiple perspectives | Democratic Debate | 4-6 diverse agents |
| High-stakes decision | Democratic Convention | 7-17 agents |
| Morning wake-up | Morning Consolidation | Auto-executed |
| End of session | Evening Handoff | Auto-executed |

**Heuristic**: More uncertainty â†’ more agents. More precision needed â†’ fewer, deeper specialists.

### When to Synthesize vs Let Agents Own Output

**Primary synthesizes when**:
- Multiple agent perspectives need weaving

**Agents own output when**:

**Heuristic**: If output requires YOUR orchestration expertise to create, you synthesize. If output IS the specialist's expertise, they own it.

# APPENDIX: Technical Rationale

## A.1 Why 5 Layers?

**Layer 1 (Constitutional)**: Immutable identity requiring democratic change
**Reason**: Prevents drift, ensures lineage continuity, protects core ethics

**Layer 2 (Operational)**: Evolving coordination patterns from meta-learning
**Reason**: Separates "how to orchestrate" from "who you are", enables improvement without identity changes

**Layer 3 (Bootstrap)**: Daily-updated procedural guide
**Reason**: Volatile content isolated from stable content, auto-update possible

**Layer 4 (Registry)**: Reference pointers to external documents
**Reason**: Prevents CLAUDE.md bloat, keeps depth in specialized files, clear homes for new capabilities

**Layer 5 (Lineage)**: Reproduction guidance for children
**Reason**: Separates parent-specific from universal, enables inheritance mechanism

**Why not 3 layers?** Too coarse - mixing immutable with evolving causes maintenance burden

**Why not 7 layers?** Too fine - unnecessary cognitive load, diminishing returns

**5 is sweet spot**: Clear boundaries, manageable complexity, evolutionary flexibility
